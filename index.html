Purpose & Scope

Purpose

This High-Level Design (HLD) defines the secure network architecture for an academic research institution engaged in advanced, government-funded and defense-related research.

  • Protection of Research Data and Intellectual Property (IP)
  • Controlled collaboration with academic and technical partners in Israel and USA
  • Addresses supply-chain risk, insider threat, AI-driven zero-day exposure, and ransomware

In Scope

  • ZTNA/SASE user access for Staff/Students/IT and Research users
  • Segmentation into Users, Research, Internal Services, OOB Management, Physical Security, Backup, and DR zones

Out of Scope

  • Secure Web Gateway for Internet egress control and inspection
  • Supplier connectivity via Suppliers FW with explicit allowlists

Business & Technical Requirements

Business Requirements

Availability 99.9% for user-facing services; 99.99% for security/control-plane
Performance Low-latency access for internal apps; predictable web access via SWG
Scalability Support growth in users and R&D workloads; add zones/services without redesign
Compliance Align with ISO 27001 style controls; data handling by geography

Technical Requirements

Bandwidth Sufficient for web breakout and R&D data transfers; DR replication capacity
Latency <50ms intra-site; ZTNA access optimized; minimize inspection on latency-sensitive flows
Security North-south and east-west firewalls; SWG; OOB management; SIEM logging
Remote Access ZTNA/SASE with role-based policy; MFA via IdP; no direct inbound access
Automation Config-as-code for FW/SWG policies; standardized templates

Design Principles & Architecture

Design Principles

  • High Availability: Redundant components, HA pairs for firewalls, dual ISP connectivity
  • Security by Design: Zero Trust access model, defense-in-depth, segmentation
  • Least Privilege: Role-based access control, explicit allowlists
  • Scalability: Modular zone design for easy expansion
  • Simplicity: Clear zone boundaries, standardized policies

Architecture Summary

Segmented, firewall-enforced zones with identity-driven access (ZTNA/SASE) for users. All north-south traffic is inspected/controlled at the edge (External FWs + Secure Web Gateways). Administrative access is isolated via an OOB zone with jump servers and SIEM. Research workloads are separated from general users with dedicated Research firewalls and a DR environment.

Threat Landscape Mitigation

  • Supply-chain: Strict segmentation and allow-listed access
  • Insider risk: Zero Trust access, least-privilege policies, full session logging
  • AI/zero-day: Isolated research environments, controlled Internet access
  • Ransomware: East-west segmentation, resilient backup networks

Network Zones

Zone Purpose
DMZ Controlled north-south edge: External FW (Users/R&D), Secure Web Gateways, Internet egress/ingress, Repos exposure as required
Internal Core data networks: Domain Servers, Internal Web Servers, R&D Web Servers, App Servers, AI Servers
Management OOB (Out-of-Band) management zone: OOB FW, Jump Servers (bastion), Infosec Servers, SIEM; includes backup networks and admin-only access paths
User User access segment(s) for Staff/Students/IT; access mediated via ZTNA/SASE; local campus LAN/Wi‑Fi aggregation
Research Isolated R&D workloads (AI/HPC, Web/App servers, Repos); tightly controlled access and Internet egress
Other Suppliers/third-party connectivity; Physical Security zone; DR Zone mirroring critical R&D services

Security Controls

Endpoint Protection

  • EDR: Advanced threat detection and response
  • DLP: Data Loss Prevention on all endpoints
  • Full-Disk Encryption: Mandatory on all managed devices
  • MDR: 24/7 managed detection and response

Network Security

  • Firewalls: North-south and east-west enforcement
  • IDS/IPS: NGFW IPS on edge and high-risk boundaries
  • WAF: Provided by SWG for outbound; consider dedicated for inbound
  • DDoS: ISP/cloud-based protection and rate-limiting

Physical Security

  • Isolated Network: Physical Security zone fully isolated from Internet
  • OOB Access: Connected only to OOB management network
  • CCTV & Access Control: Integrated monitoring systems

Access Control

  • Central IdP: MFA/SSO integrated with ZTNA/SASE
  • Role-Based Access: By population (Staff/Students/IT/Research) with least privilege
  • Zero Trust: No direct inbound access to internal zones
  • OOB Jump Servers: Bastion hosts with MFA requirement
  • Device Authentication: TACACS+/RADIUS for network devices

Encryption

  • In-Transit: TLS for all application traffic; encrypted admin protocols
  • At-Rest: Full-disk encryption on endpoints; backup encryption
  • TLS Inspection: Partial (SWG for high-risk categories; bypass for privacy-sensitive destinations)
  • ZTNA Tunnels: Encrypted from endpoints to SASE

High Availability & Resiliency

Component HA Design
Firewalls HA pairs (Active/Standby or Active/Active) per edge: Users FW, DC FW Users, DC FW Research, Research User FW, OOB FW, Suppliers FW; sync policies/state where supported
Core/Edge Routers Redundant edge routers/switches; dual uplinks from key zones to core; dynamic routing where appropriate
Load Balancers If required for web/app tiers, deploy redundant load balancers in R&D/Internal zones
Internet/WAN Links Dual ISP or diverse paths for Internet egress; separate paths for supplier connectivity if mission-critical
Cloud Connectivity Redundant tunnels/links to ZTNA/SASE and IdP; monitor SLA and failover behavior
DR Zone Mirrored Research services; automated replication and failover capability; tested recovery procedures

Risks, Open Questions & Summary

Key Risks & Mitigations

  • Supply-chain compromise: Supplier segmentation, explicit allow-lists, continuous monitoring
  • Insider threat: ZTNA-based access, role separation, SIEM correlation
  • Zero-day exploits: Isolated research zones, controlled Internet access, sandboxing
  • Ransomware: Email/web security, EDR containment, east-west segmentation, immutable backups
  • Physical intrusion: Encrypted endpoints, strong access controls, CCTV monitoring

Open Questions

  • Firewalls/SWG platforms support required policy granularity, logging, and HA?
  • TLS inspection eligibility due to privacy/compliance constraints
  • Supplier connectivity and data residency requirements
  • ZTNA/SASE and IdP services SLA confirmation
